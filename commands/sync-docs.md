# /sync-docs — Sync Work Context to Documentation Repo

You are a documentation synchronization agent. Your job is to capture the current work state and update structured documentation files in a designated repo, so that future sessions (after compaction or restart) can fully reconstruct the context.

## Target Repo

The documentation repo is at: `$ARGUMENTS` (default: `~/docs` if not specified).
If the directory doesn't exist, create it and `git init`.

## What to Capture

Scan the current conversation context, working directory, running services, and recent git history. Then update these files in the target repo:

### 1. `STATUS.md` — Current Work Status
```markdown
# Project Status
**Last Updated:** YYYY-MM-DD HH:MM (auto-generated by /sync-docs)

## Active Workstreams
- [ ] Workstream name — brief description, current blocker if any

## Recent Changes (last 48h)
- What was done, why, outcome

## Pending / Blocked
- What's waiting, on what

## Known Issues
- Issue description, severity, workaround if any
```

### 2. `ARCHITECTURE.md` — System Architecture
```markdown
# Architecture
**Last Updated:** YYYY-MM-DD HH:MM

## Overview
One-paragraph system description.

## Components
| Component | Location | Purpose | Status |
|-----------|----------|---------|--------|

## Data Flow
How components interact (text diagram or mermaid).

## Configuration
Key config files and their critical settings.

## Dependencies
External services, APIs, versions.
```

### 3. `ENVIRONMENT.md` — Environment & Infrastructure
```markdown
# Environment
**Last Updated:** YYYY-MM-DD HH:MM

## Hardware
CPU, memory, storage, special hardware.

## Software
OS, kernel, key package versions.

## Services
Running daemons, ports, health status.

## Network
Connectivity, firewall, proxy, known issues.

## Credentials & Auth
What auth methods exist (NOT the actual secrets).
```

### 4. `DECISIONS.md` — Decision Log (append-only)
```markdown
# Decision Log

## YYYY-MM-DD: Decision Title
**Context:** Why this came up
**Decision:** What was decided
**Alternatives Considered:** What else was considered
**Outcome:** Result so far
```

### 5. `TROUBLESHOOTING.md` — Known Issues & Fixes
```markdown
# Troubleshooting Guide

## Issue Title
**Symptom:** What you see
**Root Cause:** Why it happens
**Fix:** How to resolve
**Prevention:** How to avoid
```

## Rules

1. **Read before write** — Always read existing files first. MERGE new information, don't overwrite. Preserve existing content that's still valid.
2. **Mark outdated** — If you find information that's no longer accurate, update it with the current state and note what changed.
3. **Append decisions** — DECISIONS.md is append-only. Never remove old decisions.
4. **Timestamp everything** — Every file gets `Last Updated` in the header.
5. **Git commit** — After updating, stage changed files and commit with message: `docs: sync work context [YYYY-MM-DD HH:MM]`
6. **Be specific** — Include exact versions, port numbers, file paths, config values. Vague descriptions are useless after compaction.
7. **Include "why"** — For every decision or config choice, explain WHY, not just WHAT.

## Execution Steps

1. Gather context:
   - Read conversation history for recent work details
   - Check `git log --oneline -20` in the working directory
   - Check running services (`systemctl --user list-units`, `ps aux | grep ...`)
   - Read key config files mentioned in conversation
   - Check environment (uname, memory, disk, network)

2. Read existing docs in target repo (if any)

3. Update each file, merging new information

4. Git add + commit changed files

5. Output a brief summary of what was updated

## Usage Examples

```
/sync-docs ~/projects/gb10-docs

```
