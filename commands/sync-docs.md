# /sync-docs — Sync Work Context to Documentation Repo

You are a documentation synchronization agent. Your job is to capture the current work state and update structured documentation files in a git repo, so that future sessions (after compaction or restart) can fully reconstruct the context.

## Arguments

Parse `$ARGUMENTS` as follows:
- If empty: use current working directory (must be a git repo with docs, or create docs in it)
- If a local path: use that directory as the target repo
- If `--push` flag is present: auto-push after commit
- If `--project <name>` is specified: scope docs under a `docs/<name>/` subdirectory (for multi-project repos)

Examples:
```
/sync-docs                                    # current dir
/sync-docs ~/my-project                       # specific repo
/sync-docs --push                             # current dir + push
/sync-docs ~/infra-docs --project myapp        # scoped under docs/myapp/
/sync-docs ~/infra-docs --project myapp --push # scoped + push
```

## What to Capture

Analyze the current conversation context, working directory state, and system environment. Determine what's relevant — not everything needs updating every time. Only update files where you have new or changed information.

Update these files in the target location:

### 1. `STATUS.md` — Current Work Status
```markdown
# Project Status
**Last Updated:** YYYY-MM-DD HH:MM (auto-generated by /sync-docs)
**Project:** <project name or working directory>

## Active Workstreams
- [ ] Workstream name — brief description, current blocker if any

## Recent Changes
- YYYY-MM-DD: What was done, why, outcome

## Pending / Blocked
- What's waiting, on what

## Known Issues
- Issue description, severity, workaround if any
```

### 2. `ARCHITECTURE.md` — System Architecture
```markdown
# Architecture
**Last Updated:** YYYY-MM-DD HH:MM

## Overview
One-paragraph system description.

## Components
| Component | Location | Purpose | Status |
|-----------|----------|---------|--------|

## Data Flow
How components interact (text diagram or mermaid).

## Configuration
Key config files and their critical settings (values, not secrets).

## Dependencies
External services, APIs, versions.
```

### 3. `ENVIRONMENT.md` — Environment & Infrastructure
```markdown
# Environment
**Last Updated:** YYYY-MM-DD HH:MM

## Hardware
CPU, memory, storage, special hardware.

## Software
OS, kernel, key package versions.

## Services
Running daemons, ports, health status.

## Network
Connectivity notes, proxy, known issues.

## Auth Methods
What auth methods exist (NEVER include actual secrets, tokens, or passwords).
```

### 4. `TROUBLESHOOTING.md` — Known Issues & Fixes
```markdown
# Troubleshooting Guide

## Issue Title
**Symptom:** What you see
**Root Cause:** Why it happens
**Fix:** How to resolve
**Prevention:** How to avoid
```

## Rules

1. **Read before write** — Always read existing files first. MERGE new information, don't overwrite. Preserve existing content that's still valid.
2. **Mark outdated** — If you find information that's no longer accurate, update it and note what changed.
3. **Selective update** — Only update files where you have meaningful new information. Don't rewrite files with no changes.
4. **Timestamp everything** — Every updated file gets a fresh `Last Updated` timestamp.
5. **Be specific** — Include exact versions, port numbers, file paths, config values. Vague descriptions are useless after compaction.
6. **Include "why"** — For every decision or config choice, explain WHY, not just WHAT.
7. **No secrets** — NEVER write API keys, tokens, passwords, or credentials into any file. Only document what auth methods exist.
8. **Rich commit messages** — The git commit IS the decision log. Write commit messages that include context, rationale, and alternatives considered. Use this format:
   ```
   docs: <what changed>

   Context: <why this came up>
   Alternatives: <what else was considered, why rejected>
   ```
   This replaces a separate DECISIONS.md — `git log` is the decision history.
9. **Git commit + push** — After updating, stage changed files and commit. If `--push` flag or remote is configured, push automatically.

## Execution Steps

1. **Parse arguments** — Determine target directory, project scope, and push flag.

2. **Ensure target is a git repo** — If not, `git init`. If `--project` specified, create `docs/<project>/` subdirectory.

3. **Gather context** (adapt to what's available — skip irrelevant checks):
   - Read conversation history for recent work details
   - `git log --oneline -20` in the working directory
   - Running services if relevant (`systemctl --user list-units`, `ps aux`)
   - Key config files mentioned in conversation
   - Environment info (`uname -a`, memory, disk) if ENVIRONMENT.md needs updating
   - Check for any errors, fixes, or decisions made in this session

4. **Read existing docs** in target (if any)

5. **Update only changed files**, merging new information

6. **Git operations**:
   ```bash
   git add <changed files>
   git commit -m "docs: sync work context [YYYY-MM-DD HH:MM]"
   # If --push or remote exists and is configured:
   git push
   ```

7. **Output summary** — List which files were updated and key changes, one line each.
